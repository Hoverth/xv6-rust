The virtual memory
## Page Hardware

Before we introduce our virtual memory mapping implementation, we first introduce the RISC-V hardware standard for paging.  

Our OS runs on Sv39 RISC-V, and for virtual addresses running at 64-bit using only their low 39-bit, 25-bit is unused; the virtual address uses a high 27-bit indexed page table in the low 39 bits to find the page table entry (`PTE`: Page Entry Table). Each page table item contains a 44-bit physical page number (`PPN`) and some logo bits.  

When the OS finds the `PPN` through the `PTE`, it searches for its physical page number according to its `PPN`, and then finds the corresponding PTE` to extract its `PPN' according to the 2-level index. After passing the three-level index, we find the final physical page number and stitches it into a true physical address with the low 12-bit offset of our virtual address.

![](../static/vm_1.jpg)

Each page table is described by 4096 bytes, which contains 512 page table entries; each page table item contains a physical address of the lower physical page number. The top page table is present as a physical address in the register for a query when the virtual address is converted.

![](../static/vm_3.jpg)

To tell the hardware to use the page table mapping mechanism, the kernel must write the physical address of the root page to the `satp` register. Each CPU has its own `satp`. Each CPU uses its own page table to direct to convert all the addresses generated by subsequent instructions. Since each CPU has its own `satp`, different CPUs can run different processes, each process has a dedicated address space described by its own page table.

Physical memory refers to the storage unit in the DRAM. Each byte in physical memory has an address (usually byte-byte address), called a physical address. The instruction uses only the virtual address, and the paging hardware is responsible for converting it into a physical address, and then sending it to the DRAM hardware to read or write to the amount of the amount you exist for. Unlike physical memory and virtual addresses, virtual memory is not a physical object, but a collection of abstractions and mechanisms provided by the kernel, which is used to manage physical memory and virtual addresses.

## Kernel address space

`xv6-rust` maintains a page table for each process to describe the user address space of each process. In addition, there is a single page table to describe the address space of the kernel. The kernel configures the layout of its address space to enable it to access physical memory and various hardware resources with predictable virtual addresses.

![](../static/vm_4.svg)

As shown in the figure above, the address memory space distribution of `xv6-rust`.

QEMU can simulate a machine containing RAM (physical memory) that starts at 0x80000000 with a physical address and extends to at least 0x8640000000, which we call PHYSTOP. The QEMU simulation also includes I/O devices, such as disk interfaces. QEMU provides the software interface to the software in the form of a memory-mapped control register below 0x80000000 in physical address space.

For example, the kernel itself is located in both the virtual address space and the physical memory at KERNBASE = 0x80000000. The direct mapping simplifies the kernel code for reading or writing to the physical memory. For example, when fork allocates the user memory for the child process, the assoctator returns the physical address of that memory; ifk copies the user memory of the parent process to the child process, the address is used as the virtual address.

There are several kernel virtual addresses that are not directly mapped:

The trampoline page. It maps at the top of the virtual address space. The user page table has the same mapping. Note that a physical page (including trampoline code) is mapped twice in the virtual address space of the kernel: once at the top of the virtual address space, and once directly.
Each process has its own kernel stack, which is mapped to a higher location, so `xv6-rust` can leave an unmapped protection page below it. The protection page's `PteFlags` is invalid (i.e., `PteFlags::V` input 0), so if the kernel overflow kernel stack is likely to cause an exception, and the kernel will have an emergency. If there is no protection page, the overflow stack will overwrite other kernel memory, resulting in the wrong operation, and the system is likely to crash.

When the kernel uses its stack through high-memory mapping, the kernel can also access them through directly mapped addresses. The alternative design may only have a direct mapping and use the stack at the directly mapped address. However, under this arrangement, the protection page will involve unmapping the virtual address, otherwise these virtual addresses will refer to physical memory, so it will be difficult to use.

The kernel maps trampoline's pages and kernel text by placing the flag bits `PteFlags::R`(readable) and `PteFlags::X`(executable). The kernel reads and executes instructions from these pages. The kernel maps other pages by the signage bits `PteFlags::R`(readable) and `PteFlags:W`(writeable) so that it can read and write to memory in those pages.
